# KeeperHub Builder Feedback

Built by Masa during ETHGlobal Open Agents 2026, integrating KeeperHub into **Sayonara Switch** — a permissionless dead man's switch where a KeeperHub workflow polls on-chain heartbeats and calls `execute(user)` when a user goes silent past their timeout.

This file captures the friction and wins from a first-time integration. Specific, reproducible, actionable — per the bounty rules.

---

## What worked beautifully

1. **`ai_generate_workflow` is the headline feature.** The whole reason this project picked KeeperHub was that the workflow could be born from a single English prompt at submission time. For a hackathon judge, the line "the workflow itself was generated by Claude from a paragraph of English" is the demo highlight.
2. **Remote MCP via OAuth.** One `claude mcp add` line and the browser auth flow worked first try. No keys to paste, no `.env` for the demo machine. Better DX than 90% of MCP servers I've connected.
3. **Permissionless execution model fits.** Because KeeperHub agentic wallets can call arbitrary `web3/write-contract` actions, my contract didn't need a "keeper role" or whitelist. KeeperHub is just the polite default; anyone can step in.

## Friction points (specific & reproducible)

### F1 — `ai_generate_workflow` field-mapping for tuple returns

When the contract view returns a struct (`wills(address)` returns 7 fields), it took two prompt iterations to get Claude to extract the right element. The MCP tool description should mention how tuple-return outputs are referenced (e.g. is it `{{@N.result.heir}}`, `{{@N.heir}}`, or `{{@N:Label.0}}`?). Without an example in the tool docs, the LLM guesses.

**Repro**: prompt `ai_generate_workflow` to "read wills(user) and use the heir field for the next step". Observe: in the generated workflow the field reference is left as a placeholder `<heir>` in ~50% of attempts.

**Suggested fix**: include a tuple-return example in the `ai_generate_workflow` tool's `description`. Something like `"For struct returns, reference fields by name: {{@nodeId.result.fieldName}}"`.

### F2 — `web3/write-contract` revert messages are swallowed

When `execute(user)` reverted because the user wasn't actually expired yet (race during demo testing), `get_execution_logs` returned only `transaction reverted` — no decoded custom error name. My contract uses `error StillAlive(uint64 secondsRemaining)`; the selector and remaining time were lost.

**Repro**: deploy any contract with custom errors, call its reverting function via `web3/write-contract`, fetch logs.

**Suggested fix**: decode 4-byte error selectors against the contract ABI when an ABI is attached to the integration; even just surfacing the raw 4-byte prefix would be enough for a developer to grep for.

### F3 — Schedule trigger minimum is unclear

The docs imply schedule triggers can run at any cadence. I picked 60 seconds for the demo. The workflow fires reliably, but I never found explicit documentation that 60s is the minimum (or whether sub-minute is allowed). For a "fire within ~10 seconds of expiry" demo I'd want to know.

**Suggested fix**: in the schedule-trigger config UI, show the minimum interval next to the field, and surface the same in the MCP `list_action_schemas` output.

### F4 — `tools_documentation` for parameters of generated steps

After `ai_generate_workflow` produced a workflow, I wanted to see *which* fields each step references. `get_workflow` returns the JSON, but reading nested template references is hard. A `describe_workflow` companion to `tools_documentation` that produced a human-readable explanation would close the loop ("step 5 calls `execute(arg=item from step 2)`").

### F5 — Agentic wallet funding has no in-flow gas warning

When I commit a will and the KeeperHub workflow tries to call `execute()` later, I want to know *now* whether the wallet has enough Base Sepolia ETH for the gas budget. If the wallet runs dry, the workflow silently retries forever. A pre-execution dry-run that estimates gas + checks the wallet balance, exposed as `validate_workflow_runnable`, would catch this.

## Documentation gaps I hit

- **No quickstart for "KeeperHub from a Next.js app"**: the docs cover MCP and CLI but not "how do I read whether a workflow has run from my frontend". I ended up just polling on-chain `Sayonara` events instead, which is fine but means KeeperHub is invisible from the user's view.
- **Schedule + per-row execution pattern**: the canonical pattern of "schedule fires → for-each over a list of addresses → branch per item" is exactly what production keepers do. A template for it would have saved 30 minutes.
- **What does `validate_plugin_config` do *before* committing?** I called it once and got back what looked like a no-op success even when the config was clearly wrong. Needs a worked example.

## Feature requests, ranked

1. **`watch_event` trigger** — instead of polling `isExpired`, let the workflow trigger on the contract's `Heartbeat` event firing too long ago. This would halve gas usage across the network and is the natural fit for "react to on-chain state".
2. **Conditional re-arm** — let a workflow disable itself after success (so `execute()` never gets retried after the will has executed). Currently I set `executed=true` in the contract and rely on `isExpired` returning false; an in-workflow `disable_self` action would be cleaner.
3. **Per-user workflow templating** — at signup time I'd love to call something like `instantiate_template(template_id, vars={user: 0xabc})` and get a fresh workflow for that user, all from the frontend. Currently I have one workflow watching a list, which has to know the list ahead of time.
