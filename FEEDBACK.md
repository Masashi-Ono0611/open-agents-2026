# KeeperHub Builder Feedback

Built by Masa during ETHGlobal Open Agents 2026.

The submission for the main track is **Omikuji Hub** — a Claude Code Skill that turns one English prompt into a tiered USDC reward via the KeeperHub agentic wallet on Base Sepolia. See [`SUBMISSION.md`](./SUBMISSION.md). This file captures the friction and wins from integrating KeeperHub end-to-end as a solo developer over the hackathon. Specific, reproducible, actionable — per the bounty rules.

The first half of the build explored a more elaborate "watch a contract and call `execute()` on a schedule" workflow (the `app/keepersake/` directory). That exploration is what surfaced the workflow-side friction listed below; the final shipped submission narrowed scope to two MCP tools (`execute_transfer` + `get_direct_execution_status`) so that the agent runtime alone could be the entire product.

---

## What worked beautifully

1. **`ai_generate_workflow` is the headline feature.** The whole reason this project picked KeeperHub was that workflows could be born from a single English prompt at submission time. For a hackathon judge, the line "the workflow itself was generated by Claude from a paragraph of English" is the demo highlight.
2. **Remote MCP via OAuth.** One `claude mcp add` line and the browser auth flow worked first try. No keys to paste, no `.env` for the demo machine. Better DX than 90% of MCP servers I've connected. This single piece of UX is what made a same-day "prompt → tx" Skill viable.
3. **`execute_transfer` is the perfect 2-MCP-call primitive.** Synchronous response (`{ executionId, status: "completed" }`) when the tx settles fast, plus a follow-up `get_direct_execution_status` that returns the actual `transactionHash` and `transactionLink`. Two calls, ~9 seconds wall time, real Base Sepolia tx. This is what made Omikuji Hub possible without any custom contract — KeeperHub *is* the backend.
4. **Permissionless execution model fits.** Because KeeperHub agentic wallets can call arbitrary `web3/write-contract` actions, downstream contracts don't need a "keeper role" or whitelist. KeeperHub is just the polite default; anyone can step in.

## Friction points (specific & reproducible)

### F1 — `ai_generate_workflow` field-mapping for tuple returns

When the contract view returns a struct (during the keepersake exploration, `wills(address)` returned 7 fields), it took two prompt iterations to get Claude to extract the right element. The MCP tool description should mention how tuple-return outputs are referenced (e.g. is it `{{@N.result.heir}}`, `{{@N.heir}}`, or `{{@N:Label.0}}`?). Without an example in the tool docs, the LLM guesses.

**Repro**: prompt `ai_generate_workflow` to "read wills(user) and use the heir field for the next step". Observe: in the generated workflow the field reference is left as a placeholder `<heir>` in ~50% of attempts.

**Suggested fix**: include a tuple-return example in the `ai_generate_workflow` tool's `description`. Something like `"For struct returns, reference fields by name: {{@nodeId.result.fieldName}}"`.

### F2 — `web3/write-contract` revert messages are swallowed

When `execute(user)` reverted because the user wasn't actually expired yet (race during demo testing), `get_execution_logs` returned only `transaction reverted` — no decoded custom error name. My contract used `error StillAlive(uint64 secondsRemaining)`; the selector and remaining time were lost.

**Repro**: deploy any contract with custom errors, call its reverting function via `web3/write-contract`, fetch logs.

**Suggested fix**: decode 4-byte error selectors against the contract ABI when an ABI is attached to the integration; even just surfacing the raw 4-byte prefix would be enough for a developer to grep for.

### F3 — Schedule trigger minimum is unclear

The docs imply schedule triggers can run at any cadence. I picked 60 seconds for the demo. The workflow fires reliably, but I never found explicit documentation that 60s is the minimum (or whether sub-minute is allowed). For a "fire within ~10 seconds of expiry" demo I'd want to know.

**Suggested fix**: in the schedule-trigger config UI, show the minimum interval next to the field, and surface the same in the MCP `list_action_schemas` output.

### F4 — `tools_documentation` for parameters of generated steps

After `ai_generate_workflow` produced a workflow, I wanted to see *which* fields each step references. `get_workflow` returns the JSON, but reading nested template references is hard. A `describe_workflow` companion to `tools_documentation` that produced a human-readable explanation would close the loop ("step 5 calls `execute(arg=item from step 2)`").

### F5 — Agentic wallet funding has no in-flow gas warning

When the workflow tries to call `execute()` (or, for Omikuji Hub, `transfer`) later, I want to know *now* whether the wallet has enough Base Sepolia ETH for the gas budget. If the wallet runs dry, the workflow silently retries forever. A pre-execution dry-run that estimates gas + checks the wallet balance, exposed as `validate_workflow_runnable`, would catch this.

**Omikuji Hub specifically**: at ~45,047 gas units per `transfer` and ~6.6 Mwei effective gas price on Base Sepolia, each pull costs roughly 0.0000003 ETH. A `get_wallet_integration` response that included `nativeBalance`, `pendingTxCount`, and a derived "remaining executions at this gas profile" field would let the Skill warn the user before drain.

### F6 — Contract address is duplicated between frontend and workflow

A keepersake-style integration has the deployed vault address in **at least three places**: the frontend, the KeeperHub workflow's read/write nodes, and any seed scripts. Redeploying the contract (which I did once during this build, after renaming `SayonaraVault` → `KeeperSakeVault`) silently breaks the workflow because nothing tells it the address moved — the polls just return `false` forever against a dead contract.

**Repro**: deploy v1 of a contract, wire a workflow that calls one of its functions, redeploy at a new address, update the frontend, leave the workflow alone. Workflow keeps running with no errors and never fires the action.

**Suggested fix**: an **address book** primitive (per-org, per-network) — workflows reference `{{addressBook.KEEPERSAKE_VAULT}}` instead of a literal address. Updating the address book entry instantly re-targets every workflow that used it. Bonus: a CLI `kh address-book set --network base-sepolia KEEPERSAKE_VAULT 0x…` so a `bun run deploy:baseSepolia` script can pipe its output straight in.

### F7 — No "fire once" / dry-run for a deployed workflow

After generating a workflow with `ai_generate_workflow`, the only way I had to verify it works was wait for the next scheduled tick. For a 60-second schedule that's tolerable; for the 1-day production schedule it's a 24-hour feedback loop. The MCP exposes `execute_workflow` (manual trigger), but I couldn't tell from the docs whether that bypasses the trigger condition entirely or just simulates it — the difference matters when the action is `web3/write-contract`.

**Suggested fix**: `dry_run_workflow(workflow_id, simulate_inputs)` MCP tool that runs every step against a fork RPC (Tenderly-style) and returns the per-step outputs without sending a real tx. Pair it with a "Test fire" button in the workflow builder that surfaces simulated logs.

### F8 — No "is the keeper actually watching?" surface from a frontend

For the keepersake exploration I built `/dashboard` to show "your KeeperSake is being watched" but I had no API surface to prove it from the frontend. I ended up displaying just the on-chain countdown — which makes KeeperHub *invisible* to the end user. They can't tell whether the auto-execution is actually wired up or whether they need to call `execute()` themselves.

**Suggested fix**: a **public read-only REST endpoint per workflow** (`GET https://app.keeperhub.com/api/workflows/<id>/status`) returning `{ enabled, lastFiredAt, nextScheduledAt, totalRuns }`. Optional CORS allowlist per workflow so a Vercel app can fetch it directly. This is what I'd put a "🟢 KeeperHub: watching, next check in 47s" badge on the dashboard — turning KeeperHub from infrastructure into a visible reliability signal for the user.

**What I shipped instead** (workaround): the `KeeperSakeDelivered` event indexes the `caller` (the address that invoked `execute()`), and the frontend listens via `useWatchContractEvent`. When the caller matches the configured KeeperHub agentic wallet address, the row renders as a green `KeeperHub delivered…` line; otherwise it renders as orange `<addr> delivered…`. That gives judges real on-chain proof that the keeper fired — but the cadence between fires has to be *simulated* in the UI because there is no public way to confirm that intermediate `read-contract` steps actually ran. Promoting those rows from synthetic to ground-truth requires F8.

### F9 (new — Omikuji Hub specific) — `execute_transfer` returns no `transactionHash` synchronously

When the tx settles fast (which is most of the time on Base Sepolia), the synchronous return of `execute_transfer` is `{ executionId, status: "completed" }` — but **no** `transactionHash`. The Skill has to make a second `get_direct_execution_status` call to get the hash and BaseScan link. For a UX-tight CLI demo where every round-trip is visible to the judge, this is a noticeable beat (~9 seconds becomes ~10).

**Suggested fix**: when `status === "completed"` is already known synchronously, include `transactionHash`, `transactionLink`, and the `result` object in the same response. `get_direct_execution_status` becomes the fallback for the actually-async case (`status: "running"`), not the default.

### F10 (new — Omikuji Hub specific) — `list_integrations` returns no balance / chain info per row

`mcp__keeperhub__list_integrations` is the right "do I have a wallet wired up?" check, but its current return is just `[{ id, name, type, isManaged, createdAt, updatedAt }]`. To know whether the wallet can actually pay (Base Sepolia ETH for gas, USDC for rewards) I'd have to call a separate `get_wallet_integration` and parse balances. For a Skill that wants to fail-fast with "the agentic wallet is out of USDC, ask the user to top up", a one-shot `list_integrations({ includeBalances: true, network: "84532" })` would save a round-trip and a brittle parse.

## Documentation gaps I hit

- **No quickstart for "KeeperHub from a Next.js app"**: the docs cover MCP and CLI but not "how do I read whether a workflow has run from my frontend".
- **Schedule + per-row execution pattern**: the canonical pattern of "schedule fires → for-each over a list of addresses → branch per item" is exactly what production keepers do. A template for it would have saved 30 minutes.
- **What does `validate_plugin_config` do *before* committing?** I called it once and got back what looked like a no-op success even when the config was clearly wrong. Needs a worked example.
- **No "Skill-only integration" recipe**: my final submission uses **zero** workflow tools — just `execute_transfer` and `get_direct_execution_status`. There is no "you don't need workflows for everything" page anywhere in the docs. A short "minimum viable KeeperHub Skill" guide would unlock a lot of agent-runtime-only patterns.

## Feature requests, ranked

1. **Sync `transactionHash` on `execute_transfer`** (F9) — single biggest UX win for any agent-runtime-driven flow. One-line response shape change.
2. **`watch_event` trigger** — instead of polling `isExpired`, let the workflow trigger on a contract event firing (or *not* firing within a window). Halves gas across the network and is the natural fit for "react to on-chain state".
3. **Address book** (F6) and **workflow status REST endpoint** (F8) — both unblock real frontend integration patterns; without them KeeperHub stays a backend-only tool.
4. **Conditional re-arm** — let a workflow disable itself after success. Currently I have to encode "delivered=true" in my own contract and rely on `isExpired` returning false; an in-workflow `disable_self` action would be cleaner.
5. **Per-user workflow templating** — `instantiate_template(template_id, vars={user: 0xabc})` for self-serve onboarding flows.
6. **`list_integrations({ includeBalances })`** (F10) — small DX polish that lets Skills fail fast with a usable error.
